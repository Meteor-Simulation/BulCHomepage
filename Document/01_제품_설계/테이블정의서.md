# BulC Homepage 테이블 정의서

> 최종 업데이트: 2026-05-07
> PostgreSQL 16
> 정의 출처: `database/init.sql` + `database/migrations/*.sql`

## 주요 변경사항

**2026-05-07 (테이블 정의서 일관성 감사 후속 정리)**
- 헤딩 보조 번호(5-1/5-2/5-3) 제거 → 6/7/8로 일반 번호 적용, 이후 항목 +3 시프트 (총 33개 → 1~33 연속 번호)
- ERD `결제 도메인`에 `promotions` 노드 추가 (이전 누락분)
- `countries`, `revoked_offline_tokens` ToC 항목에 사용 상태 비고 추가
- 마이그레이션 파일 `V002__add_license_plan_to_price_plans.sql` → `V20260212__add_license_plan_to_price_plans.sql`로 rename (날짜 기반 명명 통일)
- `init.sql`/`V20260212` 양쪽에 동기화 주의 주석 추가 (Flyway 미도입 상태에서 의도된 중복임을 명시)
- `init.sql` DROP TABLE `refresh_tokens` 중복 라인 제거

**2026-05-03 (Vite 마이그레이션 PR 기준 동기화)**
- 게시판 도메인 신규 (`posts`, `post_images`) — V20260420__create_board_tables, V20260421 어노테이션/계층 컬럼 포함
- 보안 강화 테이블 누락분 추가 — `email_verification_attempts`, `login_attempts`, `signup_tickets`
- 컬럼 갱신:
  - `users` — `language_code`, `email_verified`, `email_verified_at` 누락분 보강
  - `price_plans` — `license_plan_id` (V20260212, 구 V002)
  - `payments` — `fail_reason`, `client_ip` (V20260409)
  - `payment_details` — `toss_status`, `toss_response_summary` (V20260409)

**2026-02-13 (Redeem 코드 시스템 추가)**
- `redeem_campaigns`, `redeem_codes`, `redeem_redemptions`, `redeem_user_campaign_counters` 4개 테이블 추가
- `licenses` 테이블에 `source_type`, `source_redeem_id` 컬럼 추가

**Users PK 마이그레이션: email → UUID**
- `users.email` VARCHAR PK → `users.id` UUID PK로 변경
- 모든 FK 참조를 `user_email` → `user_id` (UUID)로 변경
- JWT 토큰의 subject를 email → userId로 변경

## 결제 흐름

1. **유저 회원가입**: `users` 테이블에 레코드 생성
2. **이메일 인증**: `email_verifications` 테이블에서 인증 코드 관리, `email_verification_attempts`로 브루트포스 방지
3. **회원가입 완료**: `signup_tickets` 1회용 티켓 발급 → 가입 처리 후 `users.email_verified = TRUE`
4. **상품 선택**: `products` + `price_plans` 조회
5. **결제 진행**: `payments` 테이블에 pending 상태로 생성 (client_ip 기록)
6. **토스페이먼츠 콜백**: `payment_details`에 결제 정보 저장 (toss_status, toss_response_summary 포함), `payments.status` → completed
7. **구독 생성**: `subscriptions` 레코드 생성
8. **라이선스 발급**: `licenses` 테이블에 라이선스 생성 (`source_type=PURCHASE`, `source_order_id` 기록)
9. **환불 요청**: `payments.status` → refunded, 환불 정보 기록

---

## 구독 자동결제 흐름

1. **카드 등록**: `billing_keys` 테이블에 빌링키 저장
2. **자동갱신 활성화**: `subscriptions.auto_renew` = TRUE, `billing_key_id` 설정
3. **다음 결제일 계산**: `subscriptions.next_billing_date` 설정
4. **스케줄러 실행**: 매일 오전 9시 갱신 대상 조회
5. **결제 요청**: `subscription_payments` 레코드 생성, 토스페이먼츠 API 호출
6. **결제 성공**: `subscription_payments.status` → SUCCESS, 구독 기간 갱신
7. **결제 실패**: `subscription_payments.status` → FAILED, 재시도 스케줄링

## 리딤 코드 흐름

1. **캠페인 생성**: 관리자가 `redeem_campaigns` 생성 (product_id, license_plan_id, seat_limit, per_user_limit 설정)
2. **코드 생성**: `redeem_codes`에 SHA-256 해시로 저장 (원본 코드는 저장 안 함, 발행 시 1회만 노출)
3. **사용자 입력**: 코드 정규화 → SHA-256 → `code_hash`로 매칭
4. **검증**:
   - 캠페인 상태(ACTIVE), 좌석 잔여(seats_used < seat_limit)
   - 코드 활성화/만료/사용 횟수
   - 사용자별 사용 횟수 (`redeem_user_campaign_counters`로 원자적 enforce)
   - 이메일 도메인 제한(allowed_email_domain) — 있다면 매칭
5. **라이선스 발급**: `licenses`에 신규 발급 (`source_type=REDEEM`, `source_redeem_id` 기록)
6. **감사 로그**: `redeem_redemptions`에 사용 이력 기록
